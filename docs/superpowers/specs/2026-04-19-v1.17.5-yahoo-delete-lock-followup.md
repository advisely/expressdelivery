# v1.17.5 — Yahoo Delete Lock Follow-Up

**Date:** 2026-04-19
**Branch:** `fix/yahoo-delete-lock` (continuation)
**Status:** Implementation complete, pending CI gate
**Predecessor:** [`2026-04-14-yahoo-delete-lock-fix-design.md`](./2026-04-14-yahoo-delete-lock-fix-design.md)

## 1. Problem Statement

User report (2026-04-19): "After opening the Prime Store end-of-season-sale email on a Yahoo account, that email cannot be deleted, and any subsequent delete on a different email becomes slow instead of instant."

This reproduces the symptom v1.17.4 was meant to eliminate. Investigation found two gaps in the v1.17.4 implementation:

1. **`markAsRead` / `markAsUnread` bypass `operationQueue`.** The v1.17.4 spec section 3.3 enumerated `deleteMessage`, `moveMessage`, and `refetchEmailBody` as in-scope for queue routing but did not list `markAsRead`/`markAsUnread`. These are user-initiated mutating IMAP STORE operations on the same class as `deleteMessage`. Without queue routing they race against queued ops at the IMAPFlow mutex level, breaking FIFO ordering.

2. **`_refetchEmailBodyLocked` violates the section 2 invariant.** The v1.17.4 spec section 2 declared: "Mailbox locks must only protect IMAP fetch and state operations, not MIME parsing or DB-heavy post-processing." Section 3.2 implemented this for `syncNewEmails` (commit `e7a02d8`) but `_refetchEmailBodyLocked` was left holding the mailbox lock across the `simpleParser` call. For a marketing email with a multi-megabyte multipart HTML body on a slow Yahoo connection, this hold can exceed several seconds — blocking every subsequently-enqueued user action.

3. **`markAllRead` (Sidebar "mark all read" context menu) bypasses the queue and uses a 10s mailbox-lock timeout** while every other queued user op uses 30s, making it the most timeout-vulnerable entry point on Yahoo.

## 2. Solution

Three changes, all on `electron/imap.ts`:

### 2.1 Route `markAsRead` / `markAsUnread` / `markAllRead` through `operationQueue`

Apply the existing `_xxxLocked` split pattern (mirroring `_deleteMessageLocked` at lines 1313-1336):

- Public method becomes a thin wrapper: `if (!ctrl?.client) return false; return ctrl.operationQueue.enqueue(() => this._xxxLocked(ctrl, ...))`.
- Private `_xxxLocked` holds the `getMailboxLock(30s)` and runs the IMAP STORE.
- Replace the silent `catch {}` with `logDebug([op] error for uid=N on mailbox: msg)` to satisfy the project's silent-failure prevention rule.

### 2.2 Two-phase `_refetchEmailBodyLocked`

Mirror the syncNewEmails two-phase pattern from commit `e7a02d8`:

- **Phase 1 (under lock):** `getMailboxLock(30s)` → `for await client.fetch(uidRange, { source })` → capture `Buffer | null` to a variable hoisted outside the try → `finally { lock.release() }`.
- **Phase 2 (outside lock):** `if (!source) return null; await simpleParser(source, …)` in a separate try/catch.

The captured `source: Buffer` is heap-owned at the moment of release; no TOCTOU window. Both phases log via `logDebug` with disambiguated tags (`[refetchEmailBody] fetch error` vs `[refetchEmailBody] parse error`) so the logs make root-cause clear if either phase fails on a real-world account.

### 2.3 Align `markAllRead` lock timeout to 30s

`markAllRead` previously used 10s — out of step with the 30s ceiling adopted in v1.17.4. Yahoo's `messageFlagsAdd('1:*', ['\\Seen'])` against a folder with thousands of messages can legitimately take longer than 10s. Aligning to 30s removes a queue-starvation surface where a non-malicious slow STORE was reported as a timeout.

## 5. Risks

| Risk | Mitigation |
|---|---|
| Newly-queued `markAsRead` could starve background sync | False — sync uses `getMailboxLock` directly, not the queue (per v1.17.4 spec §3.3). Queue serializes user actions only. |
| Fire-and-forget `markAsRead` at `main.ts:824` could now log unhandled-rejection warnings | False — call site already attaches `.catch(() => {})`. AsyncQueue also pre-attaches a no-op handler to QueueDrainedError per `asyncQueue.ts:25-29`. |
| Two-phase `refetchEmailBody` exposes parsed bytes outside the lock | No new exposure — the bytes were already in the renderer-bound return value. The lock only ever protected concurrent IMAP access, not in-memory bytes. |
| `markAllRead` 30s timeout means a user can wait 30s instead of 10s in worst case | Acceptable trade-off — v1.17.4 already accepted this for delete/move/refetch; consistency outweighs the rare worst-case wait. |
