## 1. Problem Statement

User report after testing v1.18.0:

1. **Reading-pane delete on yassine@boumiza.com**: "I clicked the delete icon in the top bar of the core content, it says success but the email is still there."
2. **Unified-inbox drag-drop**: "I have the 3 accounts selected, and I clicked on an email from another custom account and I tried to drag and drop it into a bills folder but I couldn't (forbidden icon)."

Both are pre-existing latent bugs that v1.18.0's animation / card-design changes happened to surface. Neither was a v1.18.0 regression in the code-paths we changed; the user's test plan just exercised paths they hadn't exercised before. v1.18.1 fixes both AND adds regression tests that ensure they cannot return — extracting the affected logic into pure modules makes the contract testable at the unit level.

## 2. Root Causes

### 2.1 Bug 1 — silent local-only fallback in `emails:delete`

`electron/main.ts:1078-1092` (pre-fix) had:

```ts
if (uid > 0 && sourceFolder) {
  const moved = await imapEngine.moveMessage(...);
  if (moved) {
    db.prepare('UPDATE emails SET folder_id = ? WHERE id = ?').run(trashFolder.id, emailId);
    return { success: true };
  }
}

// IMAP move failed — fallback to local-only move
db.prepare('UPDATE emails SET folder_id = ? WHERE id = ?').run(trashFolder.id, emailId);
return { success: true };
```

When `moveMessage` returned `false` (Yahoo server rejection, lock timeout, network error) the local DB was still updated to point at trash AND `success: true` was returned. The renderer toasted "deleted". On the next sync tick the IMAP server still had the email in INBOX, so `syncNewEmails` re-created the local row. To the user this looked like "delete didn't work / email came back".

The v1.17.3 changelog entry described this same pattern but the fix at the time was upstream (queue + 30s lock timeout) — the silent-fallback line was left in the handler and continued to fire on the (now rarer) failures.

### 2.2 Bug 2 — missing drop handlers on unified-mode folder rows

`src/components/Sidebar.tsx:824` (the unified-mode folder render that fires when `selectedAccountId === '__all'`) had no `draggable` / `onDragOver` / `onDragLeave` / `onDrop` attributes. Compare to the single-account render at line 870 which had all of them.

Browser default behavior for an element with no `onDragOver` is to refuse drops (forbidden cursor). The user saw forbidden everywhere in unified view, regardless of which folder they aimed at.

Even with handlers, the underlying `emails:move` IPC at `main.ts:1379` explicitly throws `'Cross-account move not allowed'` because IMAP cannot atomically move messages across accounts — it would require download-from-A, upload-to-B, delete-from-A, with multiple failure modes. So the fix is two-layered: (a) add handlers; (b) decide same-vs-cross-account up front and refuse cross-account drops with a clear toast.

## 3. Solution

### 3.1 Extract pure logic — `electron/deleteEmailLogic.ts`

The IPC handler body becomes a one-line delegation:

```ts
ipcMain.handle('emails:delete', async (_event, emailId: string) => {
  return deleteEmailLogic(db, emailId, imapEngine);
});
```

`deleteEmailLogic(db, emailId, { moveMessage, deleteMessage }) → { success, error?, permanent? }` is pure (database + injected IMAP functions, no `ipcMain`, no Electron globals). It enforces:
- **Move-to-Trash path**: if `moveMessage` returns `false` or throws → return `{ success: false, error }` and **do NOT update the local DB row**. The user sees a toast, the email stays visible, they can retry.
- **Permanent-delete-from-Trash path**: same contract — if `deleteMessage` returns `false` or throws → return `{ success: false, error }` and **do NOT delete the local row**.
- **Local-only paths** (no UID, no source folder mapping, no trash folder configured): keep the local DB update — there's no reasonable IMAP operation to attempt.

The contract is pinned by 11 vitest tests in `electron/deleteEmailLogic.test.ts`.

### 3.2 Renderer error surfacing — `ReadingPane.handleDelete`

Added an `else` branch that shows `result?.error ?? t('toast.deleteFailed')` as an error toast. `ThreadList.handleDeleteEmail` already early-exits on `success: false` (no DB row mutation, no list refresh), so the row stays visible — the toast pipeline lives in the parent `App.tsx` which already wires the error path.

### 3.3 Extract pure helper — `src/lib/canDropOnFolder.ts`

`canDropEmailsOnFolder(destAccountId, draggedEmailIds, emails) → 'allow' | 'cross-account' | 'unknown'`. Pure (no DOM, no React). Pinned by 7 vitest tests in `src/lib/canDropOnFolder.test.ts`.

### 3.4 Sidebar wiring — unified-mode drag handlers

Added to the unified-mode folder render at `Sidebar.tsx:824`:
- `data-folder-id` + `data-account-id` attributes (testable + future-proof)
- `dragOverFolderId === folder.id` className for the existing `.drag-over` style
- `onDragOver`: invokes `canDropEmailsOnFolder(acc.id, draggedEmailIds, emails)`. If `'allow'` → `e.preventDefault() + dropEffect = 'move' + setDragOverFolderId(folder.id)`. Otherwise `dropEffect = 'none'` (forbidden cursor).
- `onDragLeave`: clears `dragOverFolderId`.
- `onDrop`: re-checks the assessment (defense in depth — `dropEffect: 'none'` should prevent drop firing per spec but browsers vary). If `'allow'` → delegate to existing `handleFolderDrop`. If anything else → toast `dragDrop.crossAccountUnsupported`, clear drag state.

New i18n key `dragDrop.crossAccountUnsupported` added to en/fr/es/de.

## 5. Acceptance Criteria

## 6. Risks

| Risk | Mitigation |
|---|---|
| Renderer doesn't show the new error toast (silent failure for the user) | Both `ReadingPane.handleDelete` and the underlying IPC contract assert that `success: false` includes an `error` string. The unit tests verify this in 4 cases. |
| `canDropEmailsOnFolder` returns `'unknown'` for a real same-account drag (false negative — refuses a legitimate drop) | Treated as cross-account (refuse + toast). Toast is informative not silent, so the user can recognize the false negative and report. The 7 unit tests cover the common cases. |
| `dropEffect: 'none'` doesn't actually prevent the drop event from firing in some browsers | onDrop re-checks the assessment as defense in depth and refuses + toasts. |
| Future contributor inlines the IPC handler again | Comment block at `main.ts` explicitly warns and points to the regression test file. |
| Future contributor adds a new IPC channel that has the same silent-fallback temptation (e.g. `emails:move`, `emails:archive`) | Not addressed in this spec — the same pattern (extract → test → delegate) should be applied. `emails:move` already returns `success: moved` from `imapEngine.moveMessage` so it surfaces the failure to the renderer (no silent local fallback to begin with), which is correct. |

## 7. Out of Scope

- Cross-account move (download-A → upload-B → delete-A flow). Future enhancement; would need progress UI + atomic-failure rollback semantics + IMAP APPEND throttling.
- Replacing `emails:move`'s `Cross-account move not allowed` throw with a structured `{ success: false, error }` (current throw is caught by the renderer's try/catch and shown as a toast — not silent — so this is acceptable).
- Replacing `imapEngine.moveMessage`'s boolean return with an `{ success, error }` shape so the IPC handler can pass through the underlying server reason. Worth doing in a future round but not load-bearing for v1.18.1.
