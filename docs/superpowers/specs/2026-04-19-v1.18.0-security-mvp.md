# v1.18.0 — Security MVP (Phase 3)

**Date:** 2026-04-19
**Branch:** TBD (off `main` after v1.17.5 lands)
**Status:** Implementation complete, pending CI gate
**Predecessor:** Phase 17 / v1.17.x — full feature matrix in `docs/ROADMAP.md`

## 1. Problem Statement

User concern (2026-04-19): "If I receive a malware, virus, exploit, or honeytrap by email, I want the strongest security libraries and script protections to always scan before accepting the email and mention if clicking remote images is not safe if a flag is raised."

The current security posture has strong **isolation** (sandboxed iframe `allow-scripts` only, in-iframe CSP, DOMPurify) and strong **detection** (phishing URL heuristics in `src/lib/phishingDetector.ts`, SPF/DKIM/DMARC parsed via `electron/authResults.ts`). The gaps:

1. **Detection signals are computed but not surfaced where the user makes risky decisions.** Phishing/auth-fail/spoof signals exist in app state but the "Load remote images" banner shows the same neutral copy regardless of risk.
2. **Attachments are saved to disk with no extension or content checks.** A `vacation.jpg` containing `MZ` magic bytes (a Windows PE executable) writes silently. A `report.exe` saves without prompting beyond the OS file dialog.

This spec is the MVP slice agreed with the user — the smallest set of changes that surfaces existing detection where it matters, plus a basic attachment safety gate.

Out of scope (deferred):
- ClamAV / external AV integration
- Google SafeBrowsing / PhishTank API lookups
- Sender-persistent remote-image allowlist
- On-ingest content scan (sigma-style rule engine)
- Quarantine pipeline

## 2. Solution

Three coordinated additions:

### 2.1 Risk-aware remote-image banner

`src/components/ReadingPane.tsx` already shows a neutral "Remote images are blocked for privacy" banner with a "Load remote images" button when blocked-image count > 0. This change:

- Adds a new module `src/lib/senderRisk.ts` exporting `assessSenderRisk(email, phishingResult) → { isHighRisk, reasons }`. Combines:
  - `phishingResult.isPhishing` from the URL heuristic detector
  - `auth_dmarc`, `auth_dkim`, `auth_spf` ∈ {`fail`, `softfail`, `permerror`}
  - `detectDisplayNameSpoofing(from_name, from_email)` from `src/lib/phishingDetector.ts`
- The banner reads `senderRisk` and switches to a danger variant when `isHighRisk`:
  - Background → `rgba(--color-danger, 0.12)`, border red, role=`alert`
  - Copy → "Loading remote images is NOT recommended for this email — risk indicators were detected:"
  - Up to 2 risk reasons rendered as a `<ul>` underneath
  - Button → red `Load anyway` instead of accent-colored `Load remote images`

The behavior of the click handler is unchanged — the user can still load if they consent. The change is purely informational + visual emphasis. This is intentional: false positives must remain dismissible.

### 2.2 Attachment extension + magic-byte safety gate

New module `electron/attachmentSafety.ts` exports:

- `DANGEROUS_EXTENSIONS: ReadonlySet<string>` — denylist of 47 extensions across:
  - Windows executables: `.exe .scr .bat .cmd .com .pif .msi .lnk .dll .cpl .msc .gadget .application .appx .msix`
  - Scripting: `.vbs .vbe .js .jse .jar .ps1 .ps2 .psm1 .wsf .wsh .hta .reg .scf .url`
  - Office macro-enabled: `.docm .dotm .xlsm .xltm .pptm .potm .ppam .sldm`
  - Disk images: `.iso .img .vhd .vhdx .ace`
  - Unix shell: `.sh .bash .zsh .command`
- `detectMagicBytes(buffer) → MagicType | null` — hand-rolled detector for: PE/MZ (`exe`), ELF (`elf`), PDF (`pdf`), ZIP/Office Open XML/JAR (`zip`), PNG (`png`), JPEG (`jpeg`), GIF (`gif`), RTF (`rtf`), OLE compound document (`ole` — legacy `.doc`/`.xls`/`.ppt`), HTML (`html`).
- `assessAttachmentRisk(filename, mimeType, content) → { risk: 'safe' | 'extension' | 'mismatch', reason, detectedType? }`:
  - `extension` — filename ends with a `DANGEROUS_EXTENSIONS` entry. Always wins, no further checks needed.
  - `mismatch` — magic bytes are an executable (`exe`, `elf`) regardless of extension, OR the extension's allowlist (`ALLOWED_MAGIC_BY_EXTENSION`) does not contain the detected magic type. Catches:
    - exe/elf disguised as `.jpg`/`.pdf`/etc.
    - ZIP container disguised as `.txt`/`.png`
    - HTML phishing page disguised as `.pdf`
    - RTF (CVE-2017-0199 vector) disguised as `.pdf`
  - `safe` — otherwise.

The `electron/main.ts` `attachments:save` IPC handler invokes `assessAttachmentRisk` after base64-decoding the content. If the result is not `'safe'` and the renderer did not pass `confirmed: true`, the handler returns `{ requiresConfirmation: true, risk, reason, detectedType }` instead of writing to disk. The actual save is deferred until the renderer re-invokes with `confirmed: true`.

### 2.3 Renderer-side ConfirmDialog gate

`ReadingPane.tsx`:
- Adds `pendingRiskySave` state — when `attachments:save` returns `requiresConfirmation: true`, the renderer parks the bytes here.
- Renders a `ConfirmDialog` (existing component) with `variant="danger"`, the risk reason as the description, and a "Save anyway" confirm button. On confirm, re-invokes `attachments:save` with `confirmed: true` and the same bytes.
- Fully respects the existing toast error pipeline for genuine save failures.

## 4. Threat Model

Threat actors and what this spec defends against:

| Actor | Vector | v1.18.0 defense |
|---|---|---|
| Phishing email author | Display-name spoof + tracking-pixel remote images | Risk banner reasons enumerate spoof + auth fail; user warned before consenting |
| Malware distributor | Send `vacation.jpg` with MZ payload | `attachments:save` blocks; ConfirmDialog explicitly names "Windows EXE executable" |
| Malware distributor | Send `report.exe` directly | DANGEROUS_EXTENSIONS confirmation gate |
| Macro-virus author | Send `invoice.docm` | DANGEROUS_EXTENSIONS gate |
| Office Open XML weaponizer | Embed exploit in `report.docx` | NOT defended — `.docx` is a legitimate ZIP-based format. AV scan would be required (deferred). |
| RTF CVE-2017-0199 author | Send `invoice.pdf` containing `{\rtf1\…\objupdate}` | Magic-byte mismatch flag; ConfirmDialog warns "content is RTF" |
| Phishing landing page | Send `invoice.pdf` containing HTML phishing | Magic-byte mismatch flag |
| RTLO bidi attacker | Send `photo<U+202E>gpj.exe` (renders "photoexe.jpg") | Filename sanitized in dialog message; user sees the real `.exe` extension |

Out of threat model:
- Malicious renderer code injecting `confirmed: true` — the renderer is fully trusted in the Electron threat model.
- TOCTOU between safety check and disk write — both phases use the same in-memory buffer.
- Side-channel data exfil via remote images even when blocked — defense-in-depth via iframe CSP.

## 5. Acceptance Criteria

**Functional (manual after build):**
- Receive a phishing email (or simulate one via authResults DKIM=fail in dev): the remote-image banner shows red, lists the failing-DKIM reason, and the button reads "Load anyway".
- Receive an attachment named `test.exe`: clicking download shows ConfirmDialog with "Files with the .exe extension can run code…".
- Receive `cat.jpg` whose first bytes are `4D 5A` (PE header): clicking download shows ConfirmDialog with "is named like a benign file but its content is a EXE executable".
- Receive a real `.docx`: download proceeds with no confirmation (legitimate ZIP-based format).
- Receive a real `.png`/`.pdf`: download proceeds with no confirmation.
- The "Cancel" button on the dialog leaves the file un-saved; "Save anyway" persists it normally.

## 6. Risks

| Risk | Mitigation |
|---|---|
| False positives on legitimate `.txt` files containing scripts (e.g., dev shipping a `.txt` with HTML inside) | User can dismiss with "Save anyway"; reason text is informative not blocking. |
| New magic-byte signatures missed (e.g., MS-DOS COM, Windows shortcut binary format, MachO) | DANGEROUS_EXTENSIONS catches the most common .com / .lnk extension cases. Magic-byte coverage is extensible — add to detector + tests as new vectors appear. |
| User clicks "Save anyway" out of habit | Outside MVP scope — would require keystroke delay or two-stage confirm. Deferred. |
| Renderer can pass `confirmed: true` to skip dialog | Renderer is trusted code in Electron threat model. Email content (the actual attacker surface) has no IPC access from inside the sandboxed iframe. |

## 7. Out of Scope (Deferred Phase)

- ClamAV daemon integration for attachment AV scan
- Google SafeBrowsing / PhishTank URL reputation lookups for in-email links

- Display-name spoof banner integration (signals exist but only surface in remote-images banner today)
