# v1.18.4 — PSL-aware Brand Spoofing + Explicit Auth Badge

**Date:** 2026-04-19
**Branch:** `fix/yahoo-delete-lock` (continued)
**Status:** Implementation complete

## 1. Problem Statement

User feedback after v1.18.3:
1. **"Is the amazon.ca bug fix an overfitting case or will it apply to other senders as well?"** — sharp question. v1.18.3 only data-patched Amazon and added Twitter's `x.com` alias; Apple/Google/Microsoft/PayPal regional storefronts would still false-positive. The user accepted Path A (data expansion) initially, then escalated to Path B (structural Public Suffix List approach).
2. **"Make the SPF + DKIM + DMARC pass more explicit either in the same info bubble that is currently showing or elsewhere"** — the v1.18.0 verified/unverified pair was icon-only with hover tooltips, didn't surface `partial` or `unknown` states, and was easy to miss.

## 2. Solution

### 2.1 PSL-based brand-spoofing check (Path B)

Replaced `BRAND_DOMAINS: Map<string, readonly string[]>` (an enumerated regional list) with `BRAND_CONFIGS: Map<string, BrandConfig>` where each entry declares `{ aliases, allowAlgorithmic }`. The new helper `isLegitimateBrandDomain(hostname, brand)`:

1. Checks aliases first (exact match or subdomain of) — covers non-pattern aliases like `x.com` for Twitter and the only-allowed domains for US-only brands.
2. If `allowAlgorithmic` is true, uses `tldts.parse(hostname)` to extract the registrable domain. Accepts when `domainWithoutSuffix === brand` AND `publicSuffix` is not in our `SUSPICIOUS_TLDS` denylist.

This means:
- `amazon.ca` → registrable=`amazon.ca`, SLD=`amazon`, suffix=`ca` (safe) → ✅ accept
- `amazon.lu` → SLD=`amazon`, suffix=`lu` (safe) → ✅ accept (no enumeration needed)
- `amazon.tk` → SLD=`amazon`, suffix=`tk` (suspicious) → ❌ flag
- `amazon.com.evil.com` → registrable=`evil.com`, SLD=`evil` ≠ `amazon` → ❌ flag
- `apple-secure.tk` → registrable=`apple-secure.tk`, SLD=`apple-secure` ≠ `apple` → ❌ flag
- `chase.de` → brand=chase, `allowAlgorithmic=false`, alias `chase.com` doesn't match → ❌ flag

Both `analyzeUrl` Rule 4 and `detectDisplayNameSpoofing` Check 1 use the new helper, keeping the rule centralized.

### 2.2 Explicit SPF/DKIM/DMARC badge

New component `AuthVerificationBadge` rendered inline next to the sender email address. Always visible (never hidden), four states:

| Status | Color | Trigger |
|---|---|---|
| Verified sender | green | all 3 of SPF/DKIM/DMARC pass |
| Partially verified | amber | at least one passes, others don't |
| Unverified sender | red | none pass and at least one fails |
| Auth unknown | grey | all 3 are `none` (sender publishes no records) |

The badge shows three artefacts simultaneously:
1. Status label (color-coded text).
2. Three per-check chips: `SPF` `DKIM` `DMARC`. Each chip is colored by its individual value (green/red-strikethrough/amber/muted).
3. Tooltip on hover with the exact value of each check.

ARIA label reads the full breakdown for screen readers.

`src/lib/senderVerification.ts` is the renderer-side verification helper (mirrors `electron/authResults.ts`'s `getSenderVerification`). Avoids cross-importing Electron main code into the renderer.

## 4. Acceptance Criteria

**Functional (manual after rebuild):**
- Open the Amazon.ca order email — banner is normal info, NOT red.
- Open an email from `amazon.lu` (or any other un-enumerated Amazon storefront) — also no red banner.
- Open an email from a real phishing source on `.tk` or `.ml` — still flagged.
- Open the Amazon.ca email and look at the sender header — there is now a visible "Verified sender" green badge with `SPF` `DKIM` `DMARC` chips next to it (assuming Amazon's mail passes all three, which it usually does).
- Open an email with mixed auth results (e.g., DKIM=fail, SPF=pass, DMARC=pass) — the badge says "Partially verified" amber, with the DKIM chip rendered red and struck-through.
- Open an email from a sender with no SPF/DKIM/DMARC records (e.g., a personal mail server) — the badge says "Auth unknown" grey.

## 5. Risks

| Risk | Mitigation |
|---|---|
| `tldts` package vulnerability | Standard npm dep with active maintenance and very large user base (ad blockers depend on it). Tracked by the project's `npm audit` step. |
| PSL data drift over time | `tldts` ships its own bundled PSL snapshot; updates land via dependency updates. ICANN's public suffix list is authoritative and rarely retracts entries. |
| Brand-spoofing rule too permissive | The negative-test suite (`amazon.com.evil.com`, `amazon.tk`, US-only `chase.de`) verifies the rule still catches the important attack patterns. Adding a new monitored brand requires adding it to `BRAND_CONFIGS` AND deciding `allowAlgorithmic` correctly. |
| Auth badge takes header space | Badge is compact (~120px wide). Falls into the existing flex-wrap row so narrow viewports wrap naturally. Reduced-motion irrelevant — no animation. |
| User confusion about `partial` and `unknown` states | Tooltip and ARIA label spell out the breakdown. Translated. |

## 6. Out of Scope

- DMARC enforcement (using DKIM/SPF/DMARC fail to actually block emails before they hit the inbox). Currently we display, not enforce. Out of scope for an email client; would belong in a server-side filter.
- Auth-result-driven sender risk weighting (e.g., bumping `senderRisk` score when DKIM fails specifically vs SPF). Currently any explicit fail contributes equally.
- Auth-result history per sender (would require new schema). Could detect "sender used to pass DKIM, now fails" patterns.
