# v1.18.3 — Trusted Senders + Regional Brand Domains

**Date:** 2026-04-19
**Branch:** `fix/yahoo-delete-lock` (continued)
**Status:** Implementation complete

## 1. Problem Statement

User report after v1.18.2 testing of the v1.18.0 senderRisk feature:

1. "amazon.ca has been tagged as unsafe, why?" — false positive on a legitimate regional storefront.
2. "Add the feature to add a sender in the trusted list to not get this warning again" — user wants permanent suppression of false positives without manually patching the heuristics.

## 2. Root Cause (#1)

`src/lib/phishingDetector.ts` had:

```ts
const BRAND_DOMAINS = new Map([
    ['amazon', 'amazon.com'],
    // ...
]);

// Rule 4 — brand spoofing
for (const [brand, officialDomain] of BRAND_DOMAINS) {
    if (hostname.includes(brand) && hostname !== officialDomain && !hostname.endsWith('.' + officialDomain)) {
        score += 35;
        reasons.push(`Brand "${brand}" in URL but not the official domain`);
        break;
    }
}
```

For `amazon.ca`:
- `hostname.includes('amazon')` → true
- `hostname !== 'amazon.com'` → true
- `hostname.endsWith('.amazon.com')` → false

→ +35 spoof score. Combined with v1.18.0's `assessSenderRisk` which lifts the result into the danger banner, every Amazon order confirmation, shipping notification, and recommendation email from a non-US Amazon storefront got the red "NOT recommended" treatment.

The rule was structurally correct ("brand name in hostname but not the official domain") — the data was wrong. Amazon, Google, Microsoft, Apple, Twitter all run multiple regional / aliased domains.

## 3. Solution

### 3.1 Brand domains as lists (regional support)

`BRAND_DOMAINS` now maps each brand to a `readonly string[]` of all known official domains. A new helper `hostnameMatchesAnyBrandDomain(hostname, officialDomains)` performs the suffix-or-exact match across the list. Both `analyzeUrl` (Rule 4) and `detectDisplayNameSpoofing` (Check 1) use the helper, so the fix is centralized.

Amazon list (20 entries): `amazon.com`, `amazon.ca`, `amazon.co.uk`, `amazon.de`, `amazon.fr`, `amazon.es`, `amazon.it`, `amazon.co.jp`, `amazon.com.au`, `amazon.in`, `amazon.com.mx`, `amazon.com.br`, `amazon.nl`, `amazon.pl`, `amazon.se`, `amazon.sg`, `amazon.com.tr`, `amazon.ae`, `amazon.sa`, `amazon.eg`. Twitter additionally lists `x.com`.

The negative cases (spoofs like `amazon.com.evil.com`, `my-amazon-account.tk`) are still flagged because none of those hostnames match the suffix-or-exact pattern of any official Amazon domain.

### 3.2 Trusted-senders allowlist (user-managed escape hatch)

`electron/trustedSenders.ts` exposes a small DB-backed module:

- `getTrustedSenders(db) → Set<string>` — set of lowercased emails
- `listTrustedSenders(db) → string[]` — ordered array for the UI
- `isSenderTrusted(db, email) → boolean`
- `addTrustedSender(db, email) → updated list` — validates email shape, lowercases + trims, idempotent
- `removeTrustedSender(db, email) → updated list`

Storage is a single row in the existing `settings` table with key `trusted_senders` whose value is a JSON-encoded `string[]`. No schema migration required. Survives corrupted JSON / non-array values gracefully (returns empty list).

Four new IPC handlers in `electron/main.ts` (added to the preload allowlist):

- `trusted-senders:list`
- `trusted-senders:add(email)`
- `trusted-senders:remove(email)`
- `trusted-senders:is-trusted(email)`

`assessSenderRisk` in `src/lib/senderRisk.ts` gains an optional third arg `options?: { isTrusted?: boolean }`. When `isTrusted` is true, the function short-circuits and returns `{ isHighRisk: false, reasons: [] }` regardless of the underlying signals. The user has explicitly accepted the risk for this sender.

`ReadingPane`:
- Loads the trusted-sender allowlist on mount via `trusted-senders:list` and stores in a `Set<string>` state.
- Computes `isCurrentSenderTrusted` from `selectedEmail.from_email` against the set.
- Passes the flag to `assessSenderRisk` so the banner reflects the trust state immediately.
- Renders a "Trust this sender" button inside the danger banner (next to "Load anyway") when the sender is not yet trusted. One click → `trusted-senders:add` → refresh local set → banner switches to non-danger.

Translated en/fr/es/de.

### 3.3 Why remote-image blocking is unchanged by trust

Trust suppresses the **security warning** — it does NOT auto-load remote images. Remote-image blocking is a separate **privacy** decision (tracking pixels). A user who trusts amazon.ca for "this sender isn't a phisher" doesn't necessarily want every shipping email to immediately load tracking-bug images. The "Load remote images" / "Load anyway" button stays on the banner; the danger styling and the per-sender risk reasons are what go away when trusted.

- Open the Amazon.ca order email that previously triggered the red banner. The banner should now appear in its **normal** (yellow/info) variant — not red — because amazon.ca is no longer flagged as a phishing/spoof source.

## 6. Risks

| Risk | Mitigation |
|---|---|
| Maintenance burden of the regional Amazon list | New entries can be added as needed; the helper is centralized. List sits in the security-sensitive file `phishingDetector.ts` so future contributors see it during review. |
| Over-broad trust — user trusts a malicious sender once, no easy revoke | IPC `trusted-senders:remove` exists. UI to manage the list deferred to v1.18.4 / v1.19.0. Users can also wipe `settings` row directly via SQL (advanced). |
| Trust is global across accounts | Likely the desired behavior — sender identity is the trust unit, not (sender, recipient). Revisit if reported. |
| Settings table corruption affects trust list | `getTrustedSenders` and `listTrustedSenders` survive corrupted JSON / non-array values by returning empty (tested). Worst case: false positives reappear, no security regression. |

## 7. Out of Scope

- Settings → Privacy & Security tab to manage the trusted-sender list (view, remove). IPC handlers are ready; UI tab is a clean follow-up.
- Per-account trust scoping.
- Automatic trust based on auth_dmarc=pass + sender_verified='verified' (would reduce false positives but expand attack surface).
- Domain-level trust (`*@amazon.ca`) — currently each unique sender address must be trusted individually. Easy add if the per-sender granularity proves too tedious.
