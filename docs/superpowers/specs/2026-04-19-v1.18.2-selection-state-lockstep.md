# v1.18.2 — Selection-State Lockstep Fix

**Date:** 2026-04-19
**Branch:** `fix/yahoo-delete-lock` (continued)
**Status:** Implementation complete

## 1. Problem Statement

User report after testing v1.18.1: "I just had a situation of receiving 2 emails in yassine@boumiza.com, I was able to read both their content, but when I deleted the older and went back to the newer to read it again, I couldn't and any other email."

Pre-existing latent state-split bug in the Zustand store. Not introduced by v1.18.1's delete-fallback fix or v1.18.0's animations — the user's test plan (read A, read B, re-select A, delete A, try to re-read B) just happened to exercise a code path that nothing else had.

The user identified the root cause via code reading before reporting it. They were right.

## 2. Root Cause

`src/stores/emailStore.ts` had two independent setters:

```ts
// Pre-fix
setSelectedEmail: (selectedEmail) => set({ selectedEmail }),  // only clears email object
selectEmail: (selectedEmailId) => set({ selectedEmailId }),    // only sets id
```

Every delete/archive flow called `setSelectedEmail(null)` to clear the open email — but `selectedEmailId` was left pointing at the just-deleted message. The split state was:

- `selectedEmail === null`
- `selectedEmailId === <deleted-email-id>`

Most renderers read `selectedEmail` for content display and `selectedEmailId` only for the highlighted-row className. After delete + refresh:

- The deleted row was no longer in the rendered list, so the className lookup never matched anything visually — invisible.
- `selectedEmail === null` so the reading pane went blank.

Then on the user's next click of a different email, `handleSelectEmail(newerId)` fired:

1. `selectEmail(newerId)` set `selectedEmailId = newerId` ✓
2. Optimistic mark-as-read on the row in the array ✓
3. `await ipcInvoke('emails:read', newerId)` — *this is where the user perceived the failure*. On Yahoo, after a STORE+EXPUNGE the IMAP connection sometimes drops (server-side disconnect after destructive op). The `emails:read` IPC handler then needed to reconnect (~2-30s on Yahoo) before any body refetch could complete. With `body_html`/`body_text` already cached locally (from the FIRST read of the newer email), `needsBodyFetch` would be false and the IPC would return immediately — but the user reported the symptom on a fresh-arrived email whose body cache may have just been populated, and intermittent IMAP disconnects can produce inconsistent timing.

The split state itself doesn't directly cause the read to fail in the simplest path, but it is a real correctness bug that compounds with any other race condition. Fixing it removes a class of latent bugs and makes the renderer's selection logic deterministic.

## 3. Solution

### 3.1 Harden `setSelectedEmail`

```ts
// Post-fix
setSelectedEmail: (selectedEmail) =>
    set({ selectedEmail, selectedEmailId: selectedEmail ? selectedEmail.id : null }),
```

When called with `null`: also clears `selectedEmailId`.
When called with `EmailFull`: syncs `selectedEmailId` to the email's id.

This covers every existing `setSelectedEmail(null)` call site (11 of them across `App.tsx`, `ReadingPane.tsx`, `Sidebar.tsx`, `ThreadList.tsx`) without any of them needing a code change.

### 3.2 Add explicit `clearActiveEmail()` action

```ts
clearActiveEmail: () => set({ selectedEmail: null, selectedEmailId: null }),
```

Named-intent helper for delete/archive flows. The two highest-traffic call sites are switched to use it (`ReadingPane.handleDelete`, `ThreadList.handleDeleteEmail`), which makes the intent obvious in the diff and serves as a model for future delete-like handlers.

### 3.3 Pin the contract with regression tests

`src/stores/emailStore.test.ts` adds 4 tests (after the existing `setSelectedEmail accepts null to clear` test):

- `setSelectedEmail(emailFull)` syncs `selectedEmailId` to that email's id.
- `setSelectedEmail(null)` clears BOTH `selectedEmail` and `selectedEmailId`.
- `clearActiveEmail()` clears both fields.
- `clearActiveEmail()` does NOT touch unrelated state (multi-select set, folder).

`src/components/ThreadList.test.tsx` adds 1 integration test that mirrors the user's exact scenario:

- Setup: 2 emails (older + newer), user has older selected (`selectedEmail` populated, `selectedEmailId === older.id`).
- Action: click delete on older row. Mock `emails:delete` returns success and `emails:list` returns only the newer email. Wait for animation to settle.
- Assertion 1: BOTH `selectedEmail` and `selectedEmailId` are now null.
- Action: click the newer email row.
- Assertion 2: `emails:read` fires for newer's id.
- Assertion 3: `selectedEmailId === newer.id` AND `selectedEmail.id === newer.id`.

A future contributor who reverts either fix breaks the test by name.

## 5. Risks

| Risk | Mitigation |
|---|---|
| `setSelectedEmail(emailFull)` syncing `selectedEmailId` could race against an in-flight `selectEmail(otherId)` from a fast user click | The optimistic-update pattern in `handleSelectEmail` calls `selectEmail(emailId)` first, then awaits `emails:read`. With the new lockstep, the `setSelectedEmail(full)` after the await syncs the id to the *resolved* email's id. If the user clicked a third email mid-flight, the in-flight `setSelectedEmail` could overwrite the user's later click. This is a separate race that exists independently of the lockstep change — the regression test does NOT cover this case and it is intentionally out-of-scope; tracking as a separate issue if it surfaces. |
| `setSelectedEmail(emailFull)` overwriting `selectedEmailId` is unwanted in some edge case | All audit call sites (`grep setSelectedEmail`) pass either `null` or a full email whose id matches the current `selectedEmailId` (because `selectEmail` was called first). The lockstep is consistent with what every existing call site already intended. |

## 6. Acceptance Criteria

**Functional (manual after rebuild):**
- Open email A. Open email B. Re-open A. Delete A. Click B → reading pane shows B's content.
- Open email A. Delete A. Reading pane goes blank. Click email C from a different account in unified mode → C opens normally.
- Repeat across all 3 accounts (Gmail / yassine@boumiza.com Yahoo / custom). The fix is provider-agnostic.

- `src/stores/emailStore.test.ts`: 5 lockstep tests pass (1 updated + 4 new).

## 7. Out of Scope

- Race condition between in-flight `emails:read` and the user's next click (a different bug class — would need request-cancellation or last-wins guard).
- Yahoo-specific IMAP disconnect after STORE+EXPUNGE (separate investigation; if it still surfaces post-v1.18.2, would need on-401 reconnect tightening in `imap.ts`).
- Cross-account drag-drop (deferred — see v1.18.1 spec).
